# 判定实现

本章覆盖裁判层（Judger）如何管理与执行判定规则（Judge rules），以及与Mod/技能动态挂载规则相关的核心组件：JudgeRuleManager、DynamicJudgeRulePool 与 Mutation。本章不重复基础指南的 DSL/技能写法，而专注于判定系统的语义、生命周期和常见模式。

## 目的与设计要点

游戏在每个回合会经历一系列“判定阶段”（JudgeStage），每个阶段都有一套默认规则（例如交换攻击、取消攻击、应用效果等）。为了支持技能临时加入、覆盖或修饰这些默认规则，系统采用了：

- 把“临时规则”抽象为 Mutation（可带延迟与持续回合数）；
- 在编写技能时把规则以模板形式注册到全局池（DynamicJudgeRulePool）；
- 在技能编译/执行时把模板专化（specialize）为“属于某个技能使用者（ActorSet）的真实 Mutation”，并交由 JudgeRuleManager 管理；
- JudgeRuleManager 在每个回合构建并执行由默认规则 + 所有有效 Mutation 组成的执行流。

## 关键类型与职责

- Mutation
  - 表示单个“判定变更单元”。包含：RemainingRounds（持续回合数）、DelayRounds（延迟生效回合数）、Stage（在哪个判定阶段起作用）、RuleType（是覆盖 Override 还是修饰 Modifier）、ModifierOrder（Before / After）与实际的 JudgeRule（Action<ActorSet,ActorSet>）。
  - 在 pool 中以模板形式存在；在被专化后（Query -> Specialize）成为针对特定 ActorSet 的 Mutation。

- DynamicJudgeRulePool
  - 负责保存技能在编译时通过 DSL 注册的“模板” Mutation 列表（一个规则名对应多条模板）。
  - RegistDynamic(name, mutations)：把传入的 Mutation 列表转换成内部的 MutationPrototype 列表（包含一个 judge rule prototype：Action<ActorSet source, ActorSet player, ActorSet enemy>）。
  - Query(source, name)：针对具体 source（技能的施放方）把模板专化为真实 Mutation（每条 MutationPrototype 的 Specialize 返回一个 Action<ActorSet player, ActorSet enemy>，该 action 内部会通过 IfElseUtil 把 source 和 player/enemy 顺序修正，从而让模板实现：如果 source == player 则按 player->enemy 执行，否则按 enemy->player）。
  - 设计理由：规则模板本身是对“技能发起者的行为”进行描述，只有在专化时才能知道具体发起者是谁，从而在后续判定时能正确将 player/enemy 映射到规则执行的参数。

- JudgeRuleManager
  - 管理多个 JudgeStage，每个 Stage 用一个 StageRuleContainer 保存“核心规则、覆盖规则与修饰器”。
  - StageRuleContainer 内部保存：Override 列表、ModifiersBefore、ModifiersAfter，每条规则都封装为 RuleUnit（包含 RemainingRounds、DelayRounds、Rule）。
  - AddJudgeRule(source, name) 会：
    - 调用 DynamicJudgeRulePool.Query(source, name) 得到专化后的 Mutation 列表；
    - 把每条 Mutation 按 Stage 与 RuleType 注册到对应 StageRuleContainer（Override / ModifierBefore / ModifierAfter）；
  - GetRule() 会把每个 Stage 的 StageRuleContainer.Build() 串联起来，返回一个完整的 Action<ActorSet,ActorSet>，该 action 可在 Judger 执行时直接调用。
  - StageRuleContainer.Build() 实现细节：
    - 更新时间（TimePass）让 DelayRounds 与 RemainingRounds 递减；
    - 只有 DelayRounds 变为 0 且 RemainingRounds 也递减到 0 时，该 RuleUnit 的 Rule 会被包括到本回合实际执行列表中（Build 中的 Where(m => m.RemainingRounds == 0)）；
    - 顺序：先执行所有 Before 修饰器，再执行 Override（若有）或核心规则，最后执行 After 修饰器。

## 判定执行生命周期（一个典型流程）

1. 技能在 DSL 中调用 LinkJudgeRuleDynamic(name, mutations)：
   - DSL 把 name 加入 SourceFile 的 _mutationsOnCompile 同时把模板 mutations 注册到 DynamicJudgeRulePool（RegistDynamic）。注意：RegistDynamic 只保存模板（MutationPrototype），rule 是以 prototype（接受 source, player, enemy 的三参数）形式保存。

2. 当技能被编译为 Intent（SourceFile.Compile）且传入 Judger（Compile(judger != null)）时：
   - Compile 会把 SourceFile._mutationsOnCompile 中的每个 key 调用 judger.JudgeRuleManager.AddJudgeRule(_owner, key)。
   - AddJudgeRule 会调用 DynamicJudgeRulePool.Query(_owner, key) 得到针对 _owner 专化的 Mutation 列表，并把它们加入 JudgeRuleManager 的容器内。

3. 在 Judger 的运行（Judge）过程中，会根据阶段顺序调用 JudgeRuleManager.GetRule() 返回的 Action 并执行。每个 Stage 执行：
   - 先运行 Before 修饰器（若存在）
   - 再运行覆盖规则（Override 的最后一条覆盖所有），若无覆盖则运行核心默认规则
   - 再运行 After 修饰器

4. Mutation 的 DelayRounds/RemainingRounds 管理：
   - DelayRounds > 0 时先递减 DelayRounds 而不激活 Rule；DelayRounds 变为 0 后，下一次 Update/Build 时开始递减 RemainingRounds，RemainingRounds 为 0 时才被包含并执行。
   - 典型用法：delayRounds=1 表示“下回合开始才真正生效”，remainingRounds 用来表示持续回合数。

## 常见模式与示例

1. 转移（Swap）与延时保护（Delay-protect）
   - 转移需要把本回合对方发出的 Resolution（攻击/效果）改写为“下回合生效且交给另一方执行”。实现方式：在 RegistDynamic 保存的 prototype 中，JudgeRulePrototype 会把匹配到的 Resolution 的 DelayRounds 设为 1（或修改 Target/Source），从而欺骗 Judger 将本回合的某些分配延迟到下一回合并调换到另一方。DynamicJudgeRulePool 中的 IfElseUtil 帮助把 source 映射到 player/enemy。

2. 临时强化/替换判定
   - 例如某个技能需要在 OnAttackCanceling 阶段插入一个在第一时间生效的“自动反击”规则：可以注册一个 Mutation，Stage=OnAttackCanceling, RuleType=Modifier, ModifierOrder=Before，RemainingRounds=1，DelayRounds=0，JudgeRule 为一段创建并执行攻击分支的代码（在模板中用 source 参数来构造具体 Action）。

3. 被动式（被触发后回退）的规则
   - 如果技能在触发时需要在触发点挂上一段临时规则，而在某些条件下该规则应立即撤销或改变（例如 Lancer 的 Charge 的复杂逻辑），可以把触发和撤销逻辑组合为：一条在 OnAttackCanceling 的 Before 修饰器（即时触发自动反击并清零计数），以及一条带 delayRounds=1 且 remainingRounds=1 的 OnBegin 修饰器用来在下一回合检查并可能回滚 Charge 状态。

## 实用提示

- Mutation 的 JudgeRule 中不要直接访问“全局状态”的可变字段而期望在以后也能正确工作；优先通过传入的 source / player / enemy 来访问角色信息，这样 DynamicJudgeRulePool.Specialize 能正确把模板转为针对具体施法者的规则。
- 如果需要把“当回合产生的 Resolution”转移或延迟，请操作 TurnContext 中的相应分辨 Resolution 列表（例如 AttackResolutions、EffectResolutions、ResourceResolutions），而非直接修改 Actor/Body 状态。默认实现中许多例子都遵循这个约定。
- 小心 DelayRounds 与 RemainingRounds 的语义：DelayRounds 控制“何时开始计时”，RemainingRounds 控制“规则执行次数/持续回合”。

## 参考：源码片段（说明性）

下面为系统内重要函数的行为概述（已在主代码中实现，此处为简化说明）：

- JudgeRuleManager.AddJudgeRule 会调用 DynamicJudgeRulePool.Query(source, name)，并把返回的 Mutation 注册到对应 Stage 的容器中。
- DynamicJudgeRulePool.RegistDynamic 会把传入 Mutation 转换为内部的原型（JudgeRulePrototype 期望三个 ActorSet 参数），并存在池中以备专化。

这些实现都在项目源码中有明确定义（参考 Backend/JudgementLogic/Judgement/ 文件夹）。本节目的在于让你理解这些组件的协作与时序。
